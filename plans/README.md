# Implementation Plans

Self-contained, executor-ready implementation plans. Each plan states its own
Scope, Drift check, Steps, Verification, and STOP conditions per
`.cursor/agents/phived-plan-executor.md`. Execute in number order unless a plan's
**Depends on** says otherwise, and update its status when done.

Plans share a single zero-padded number sequence (`001`, `002`, …). The active
cloud-sync work (`007`–`010`) is listed first; the completed `001`–`006` batch
follows. Architecture decisions behind these plans live in `docs/adr/`.

Status values: TODO, IN PROGRESS, DONE, BLOCKED, REJECTED.

## Active — cloud sync (ElectricSQL + Better Auth + Polar)

Cross-device sync for paying users, added as an **additive, local-first** layer:
logged-out and non-paying use stays exactly today's local-only app. Four strictly
sequential plans; each one leaves the app shippable.

| 008  | Backend foundation (Postgres, server, Better Auth, Polar) | –          | TODO   |

Decisions behind these plans:

- `docs/adr/0004-cloud-sync-via-electricsql.md` — sync architecture
- `docs/adr/0005-auth-and-billing-for-sync.md` — auth + billing
- `CONTEXT.md` — glossary (`Accounts & sync`, `Tasks`)

### Order & why

1. **007 — Local task identity** (`apps/app` only, no backend). Give every task a
   stable `id`, separating task _identity_ from view _padding_. Ships invisibly
   (no behaviour change) and is the prerequisite for any row-level merge. The
   riskiest local change because it touches the core data model and every
   existing user's stored data.
2. **008 — Backend foundation** (Railway). Postgres, the site→app server, Better
   Auth (passwordless), Polar subscription + webhooks, and entitlement state.
   Sync is **not** wired into the app yet (flag-gated). Outcome: a user can sign
   in and subscribe; nothing syncs.
3. **009 — Electric read sync**. Private Electric service + an authenticated
   gatekeeper; entitled users stream _their own_ cloud rows into the app
   (read-path only; writes still local). Proves the pipe end-to-end.
4. **010 — Write path + merge**. The write API + optimistic/offline writes
   (TanStack DB), first-sign-in union-merge, account deletion, and lapse
   handling. Completes the sync loop.

Identity must exist before anything can merge (007). Auth + billing must exist
before sync can be gated to payers (008). Reads are the easy half and validate
the whole pipe before the harder write path (009). Writes need identity, auth,
gating, and read-sync all in place first (010).

### Invariants every phase must preserve

- **Single public origin** `phived.com`; `/sw.js` stays a real root file with
  `Cache-Control: public, max-age=0, must-revalidate`; the `/app/*` SPA fallback
  is unchanged (ADR 0001–0003, `scripts/site-contract.mjs`).
- **No regression for anonymous / non-paying use** — that path stays exactly
  today's local-only app.
- **Local data is the device's source of truth; billing state never deletes
  it.**
- **Electric is never publicly reachable.**

### Shared conventions

- Bun for everything. The full gate is `bun run check` at the repo root.
- Secrets go in Railway service variables and git-ignored `apps/app` `.env`
  files; never commit them. The exact backend code location is fixed in 008.
- Never delete `apps/web/public/sw.js`.
- **Logical replication on Railway is supported** (confirmed: Railway's official
  "Deploy ElectricSQL" template + real deployments) — not a blocker. Enable
  `wal_level=logical` via the Postgres service's Custom Start Command flags
  (`-c wal_level=logical -c max_wal_senders=10 -c max_replication_slots=10`),
  which is robust against the image's boot-time config rewrite. Details in
  008/009.
